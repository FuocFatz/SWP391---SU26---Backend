1. Project Overview
1.1 Background & Academic Context
EquiX is developed as the capstone deliverable of SWP391, the senior software engineering project course at FPT College Vietnam. The course challenges student teams to ship a complete, end-to-end web application that demonstrates mastery of full-stack development, requirements engineering, database design, security, and user experience within a single academic semester. EquiX was selected by the team as a domain that is rich enough to exercise every layer of the technology stack while remaining bounded enough to be delivered as a university-level Minimum Viable Product (MVP). The project intentionally avoids high-risk real-world concerns such as live monetary gambling, regulatory compliance, or integration with real horse-racing authorities, and instead substitutes a structured "guess" model that preserves the player-facing excitement of racing without exposing the platform to financial regulation.
The academic scope deliberately constrains the system to a self-contained tournament simulator in which every role — Horse Owner, Jockey, Referee, Spectator, and Admin — is played by a registered human user rather than an external stakeholder. This allows the team to demonstrate role-based access control, multi-actor workflows, and complex state machines in a controlled environment that a single mentor can evaluate end-to-end during a demo session. Version 4.0 of this document is the authoritative consolidation of three prior drafts and folds in the final scope decisions made after mentor consultation, including race cancellation, password recovery, notifications, audit logging, and demo-mode time compression.
1.2 Technology Stack
The technology stack was chosen to align with the team's existing skills, the course's teaching track, and the need for rapid iteration during the demo period. ReactJS provides a component-driven frontend that can be developed and hot-reloaded quickly, while Spring Boot on the Java Virtual Machine offers a mature, well-typed backend with first-class support for REST APIs, security, and transactional database access. JWT is used as the stateless authentication mechanism so that the backend can scale horizontally without sticky sessions, and MySQL is selected as the relational database for its ACID guarantees and widespread hosting support. 1.3 Theme & Design Philosophy
The visual identity of EquiX is deliberately "Modern + Easy-looking": clean surfaces, generous whitespace, flat iconography, and a restrained three-colour palette. The primary accent is a deep racing Red (#C0392B) that references traditional horse-racing silks and signals energy, urgency, and competition; it is reserved for primary actions, the brand wordmark, and critical alerts. The secondary accent is a vibrant Green (#27AE60) that represents the turf surface and stands for success, confirmation, and forward progress; it is used for confirm buttons, success toasts, and dividers. The neutral canvas is an off-white White (#FAFAFA) that reduces eye strain during long demo sessions and provides high contrast for both the red and the green accents without becoming clinical.
The colour philosophy is intentionally symbolic: red never appears on a success path and green never appears on a destructive path, so users develop reliable colour-based heuristics for navigating the system. Typography uses a single humanist sans-serif family throughout to avoid visual noise, with size and weight contrast carrying the entire information hierarchy. Every interactive element is sized for comfortable desktop use because the primary demo device is a laptop; mobile responsiveness is preserved but not optimised.
1.4 Scope Decisions
Several deliberate simplifications were made to keep the project within the SWP391 academic envelope. First, no real currency is ever handled: the prize pool is a display-only field denominated in VND, and spectator rewards are limited to in-kind goods (horse-related merchandise, voucher codes, and drink coupons) rather than cash payouts. Second, the race simulation is intentionally non-physical: positions are randomised at fixed intervals and the final finishing order is determined by a single randomisation event when the timer reaches zero, rather than by a physics or stats engine. Third, a Demo Mode is provided so that the mentor can experience the full lifecycle in minutes rather than days; this mode compresses scheduling windows and relaxes certain grace-period constraints, all clearly labelled in the UI.
Additional scope decisions include the omission of live video streaming (replaced by an animated racetrack), the omission of real payment gateways, and the simplification of the tournament concept to a single "final race" of a hypothetical prior tournament (see Section 16). These choices allow the team to demonstrate depth in business-logic correctness rather than breadth in peripheral integrations, and they keep the demo deterministic enough to be reproduced reliably during evaluation.
2. System Architecture
2.1 Three-Tier Architecture
EquiX follows a classic three-tier architecture with strict separation between presentation, business logic, and data access layers. This separation is enforced structurally: the ReactJS frontend communicates exclusively over versioned REST endpoints and never constructs raw SQL, the Spring Boot backend exposes DTOs rather than JPA entities to the outside world, and the JPA repositories are the only components permitted to touch the database. The table below summarises the responsibilities of each tier and the technologies used to enforce the boundary.

The three-tier split pays dividends during testing: each tier can be unit-tested in isolation. Frontend components are tested with React Testing Library against mocked axios responses; services are tested with JUnit and Mockito against stubbed repositories; and repositories are tested with @DataJpaTest against an embedded MySQL-compatible H2 instance. Integration tests then exercise the full stack through MockMvc to verify that contracts between tiers remain intact.
2.2 Authentication & Authorization Flow
Authentication is implemented with JSON Web Tokens using the HS256 symmetric signing algorithm. On a successful login the backend issues an access token containing the user id, role, and a short expiry (default 30 minutes), plus a longer-lived refresh token (default 7 days) that can be exchanged for a new access token without re-entering credentials. Both tokens are returned in the response body and persisted client-side in memory only — they are never written to localStorage or sessionStorage, mitigating the impact of cross-site scripting attacks. A JWT validation filter intercepts every protected request, verifies the signature and expiry, and populates the Spring Security context with the user's role so that method-level @PreAuthorize annotations can enforce authorization.
Role-based access control is enforced at three concentric layers. The outermost layer is URL-level: Spring Security configuration restricts each /api/v1/<role>/* path prefix to the matching role so that, for example, only a referee can reach /api/v1/referee/*. The middle layer is method-level: @PreAuthorize("hasRole('ADMIN')") guards individual service methods that are reachable through shared endpoints. The innermost layer is domain-level: the service itself re-checks ownership (e.g., a horse owner can only delete their own horse) so that even a compromised token for one role cannot mutate another user's data.
Account confirmation follows a tiered model based on role. Spectators are activated immediately upon registration so they can explore the platform without friction. Horse Owners and Jockeys are created in a Pending status and must be verified by an Admin before they can perform their role-specific actions; however, they may log in and complete their profile while Pending, which reduces the perception of a registration cliff. Referee accounts are created directly by an Admin rather than through the public registration form, because the referee role carries race-critical authority and the team decided that public self-signup for referees would be a security and trust risk in the MVP.
2.3 Demo Mode Adaptations
Because the SWP391 demo cannot wait days for the natural race lifecycle to unfold, the system supports a Demo Mode that compresses time without breaking the documented business rules. When Demo Mode is enabled by an Admin, race scheduling accepts start times as little as thirty minutes in the future; the one-week registration deadline is shortened to a configurable number of minutes; the five-day grace period for pair withdrawals is likewise compressed. All such compressions are surfaced in the UI via a "DEMO MODE" badge so that evaluators are not misled about the realism of the configuration. Once a race is finalized, the next race can open registration immediately, allowing the mentor to witness several complete lifecycles in a single session.
Demo Mode is implemented as a feature flag persisted in a system-settings table rather than baked into code, so that the production-grade lifecycle can be restored simply by toggling the flag off. The team explicitly considered, and rejected, the alternative of running a separate demo deployment: a single deployment with a flag keeps the demo and production code paths identical, which means any bug discovered during the demo is guaranteed to be reproducible in production and vice versa. The compressed timings are documented in Section 20 alongside the explicit list of business rules whose enforcement is relaxed while Demo Mode is active.

Three design decisions in the matrix deserve narrative justification. First, the Referee role is admin-only creation: the team decided that the referee carries exceptional responsibility (the ability to disqualify participants, to submit official reports, and to trigger Admin review loops) and that allowing public self-signup for this role would undermine the trust model of the platform. Second, Spectators are activated immediately on registration because their actions are low-risk and the platform benefits from the lowest possible barrier to engagement; the only friction is the guess lock at Standby, which is enforced structurally rather than administratively. Third, Horse Owners and Jockeys are both Pending-by-default because both roles participate in the pairing contract: an unverified owner registering a horse with an unverified jockey would create a contract that no Admin has vouched for, which the team judged unacceptable.
Cross-cutting capabilities such as viewing races, browsing leaderboards, and managing one's own profile are available to all roles, because they are read-only and reinforce shared situational awareness. Admin override rights exist throughout the matrix so that the platform can be rescued from any logical dead-end (for example, an owner who is unreachable but whose horse must be removed from a race) without resorting to direct database edits. Every Admin override is recorded in the audit log so that the override itself can be reviewed later if disputed.
4. Common Pages (Guest)
The Common Pages are accessible without authentication and constitute the public surface of the EquiX platform. They are designed to inform prospective users about the system, establish the legal framing of the platform as a "guess" rather than gambling product, and provide entry points to authentication. All Common Pages share the same global header and footer, with two prominent call-to-action buttons in the header (Login and Register) so that a curious visitor is never more than one click away from joining the platform.
4.1 Homepage
The Homepage is the landing page for unauthenticated visitors and the redirect target after logout. It opens with a hero banner displaying the EquiX wordmark in red, a tagline referencing the turf theme, and two call-to-action buttons. Below the hero, a live "Upcoming Races" carousel shows the next three scheduled races with their type, distance, and current participant count, each card deep-linking to the race detail page. A second section previews the current top of the leaderboard, and a third section explains the three role families (Owner, Jockey, Spectator) with icons and short value propositions.
The Homepage is intentionally content-light: it exists to convert visitors into registrations rather than to surface every feature. Performance budgets enforce a maximum of two API calls on initial load (upcoming races and leaderboard preview), and both responses are cached server-side for sixty seconds so that a sudden spike of demo traffic does not overwhelm the database. The page is fully responsive and degrades gracefully on narrow screens by collapsing the carousel into a single column.
4.2 About Page
The About Page tells the story of EquiX as an academic project. It introduces the team, the supervising mentor, the SWP391 course context, and the high-level goals of the system. The page also discloses the technology stack and the design philosophy described in Section 1.3, so that technically-minded visitors can understand the engineering decisions behind the product. A short FAQ at the bottom of the page answers the most common questions: "Is this real gambling?" (no), "Can I win real money?" (no, only in-kind rewards), "Who can become a referee?" (Admin-created accounts only).
From a content-governance perspective, the About Page is owned by the product team and edited through a static content file rather than the database, because the team judged that exposing it to admin-editable content would invite unnecessary risk of defacement. The page is cached aggressively at the CDN layer and is the only Common Page that does not call the backend at all on initial render, which means it loads essentially instantly even on slow connections.
4.3 Terms and Service
The Terms and Service page is a critical legal framing document. It opens with an unambiguous statement that EquiX is a university capstone project, that no real currency is handled, and that the "guess" mechanic is an engagement game rather than a gambling product. The document explicitly disclaims any prize with monetary value and clarifies that all rewards are in-kind goods whose availability is at the discretion of the platform. By registering an account, the user acknowledges that they have read and agreed to these terms, and a checkbox must be ticked on the Register page to make this acknowledgement explicit.
The Terms and Service also codify the platform's anti-abuse posture: multi-account circumvention of the "one guess per spectator per race" rule is prohibited, automated guess placement is prohibited, and any attempt to manipulate race outcomes is grounds for immediate account termination. The "Guess vs Gambling" framing is the cornerstone of the platform's legal posture in Vietnam and is repeated verbatim in the footer of every authenticated page so that users are continually reminded of the framing. The document is versioned, and the active version is recorded against each user's registration timestamp so that retroactive changes cannot bind users without consent.
4.4 Login
The Login page accepts an email and a password and submits them to the /api/v1/auth/login endpoint, which returns a JWT pair on success. Validation is performed client-side and server-side: client-side validation enforces email format and non-empty password to reduce round-trips, while server-side validation is authoritative. Error messages are deliberately generic ("Invalid email or password") to prevent user-enumeration attacks in which an attacker probes the system to discover which emails are registered. After three consecutive failed attempts from a single IP, the account is temporarily throttled for sixty seconds and the failure is logged in the audit trail.
Below the Login form, a "Forgot Password?" link initiates the password reset flow described in Section 19. The reset link is sent to the user's registered email, is valid for 15 to 30 minutes (configurable system setting, default 30 minutes), and is single-use: the moment a new password is set, the link is invalidated. Until the link is used, the old password remains active so that a forgotten reset email does not lock out a user who remembers their credentials. Security questions are optional and may be configured in the user profile as an additional recovery channel, but they are never required and are deliberately de-emphasised because of well-known weaknesses in user-chosen questions.
Successful login redirects the user to a role-specific dashboard: Horse Owners and Jockeys see their respective dashboards even while Pending, Spectators see the upcoming races list, Referees see their assigned races, and Admins see the system dashboard. A session persistence cookie is set so that a page refresh does not log the user out, but the JWT itself is held in memory only. After thirty minutes of inactivity the access token expires and the user is silently redirected to the Login page with a friendly "session expired" message rather than an abrupt 401 dialog.
4.5 Register
The Register page allows a new visitor to create one of three public roles: Horse Owner, Jockey, or Spectator. The Referee role is intentionally absent from the public form because referee accounts are Admin-created only. The form collects a common set of fields (full name, email, password, password confirmation, agreement to the Terms of Service) and then role-specific fields: Horse Owners are asked for stable name and contact phone, Jockeys are asked for career stats summary and weight, and Spectators are asked only for a display name. All passwords are hashed with BCrypt before storage, and the email is verified to be unique at registration time.
On submission, the account is created with a role-appropriate status: Spectator accounts are activated immediately and can log in and place guesses within seconds; Horse Owner and Jockey accounts are created in Pending status and must be verified by an Admin before they can perform their role-specific actions. Pending users may still log in and complete their profile so that the verification delay does not feel like a registration cliff. The pending status is clearly displayed in the user's dashboard with a banner explaining the next step, and the Admin receives a notification that a new account awaits verification. Once verified, the user receives an in-app notification and an email, and a small verified badge is displayed next to their name throughout the platform.
5. Horse Owner Business Flows
The Horse Owner is the primary producer of race content: they register horses, hire jockeys, form pairs, and submit those pairs to races. The Horse Owner experience is therefore the most complex of the five roles, spanning seven sub-areas from dashboard to personal profile. Each sub-area is described below with its own flow narrative and edge cases.
5.1 Dashboard
The Horse Owner dashboard is the landing page after login and provides a consolidated view of the owner's stable, active pairings, upcoming races, and recent activity. The top of the dashboard shows a counts strip with the number of horses owned, the number of active pairings, the number of registered race entries, and the number of pending invitations. Below the strip, a tabbed panel surfaces a list of horses (with status badges), a list of active pairings (jockey + horse + race), and a list of upcoming race entries with countdown timers. A side rail lists the most recent notifications, with deep links into the relevant detail pages.
The dashboard is designed to give the owner a one-glance answer to the question "what do I need to do next?". Pending invitations are highlighted in amber, horses whose status is Available but who have no active pairing are highlighted in blue as pairing candidates, and race entries whose race is within 24 hours are highlighted in red to signal urgency. The dashboard data is fetched through a single aggregated endpoint so that the page renders in one round-trip, and the response is cached for fifteen seconds to balance freshness against load.
5.2 Races
The Races tab lists all races visible to the Horse Owner, segmented into Upcoming, Registration Open, In Progress, and Completed. Each race card displays the race name, type, distance, scheduled start time, current participant count against the maximum, registration deadline, and prize pool. Clicking a card opens the race detail page, which shows the full participant list, the assigned referee, and a Register Pair button if the owner has an eligible pair and the registration window is open. The detail page also shows the grace-period countdown so that owners know how long they have to withdraw without Admin intervention.
5.2.1 Race Registration Flow
Registering a pair for a race is the single most rule-intensive action in the Horse Owner experience. The flow enforces a strict prerequisite chain: first, the owner must have at least one horse in the Available status; second, that horse must be paired with a jockey who has accepted the invitation (no provisional pairings are accepted); third, the resulting pair can be registered for a race only if all of the following conditions are simultaneously true.
The race has not reached its maximum participant count (max 18).
Registration is still open (i.e., the current time is before the registration deadline, which is automatically set to one week before the scheduled race start).
The chosen jockey is not already assigned to another horse for the same race (a jockey can ride at most one horse per race).
The chosen horse is not already registered for a conflicting race (a horse can be registered for at most one race whose time window overlaps this race).
The race can still potentially reach its minimum participant count of six (i.e., current registrations plus open slots make six achievable).
If any condition fails, the Register button is disabled and a tooltip explains the specific reason. The five conditions are evaluated atomically in a single transaction so that two owners racing to register the last slot cannot both succeed. On success, the pair's status becomes Registered, the horse's status becomes Registered, the jockey's status becomes Registered, and the race's participant count is incremented. A confirmation notification is sent to the owner, the jockey, and the assigned referee.
5.3 Horse Management
The Horse Management area allows the owner to perform CRUD operations on horses in their stable. Each horse has a name, breed, age, portrait, and a status field managed by the state machine described below. Horses can be added, edited, archived (soft-deleted), and restored, but a horse that is currently Paired or Registered cannot be archived until the pairing contract is dissolved. The portrait upload flow is described separately in Section 5.3.2 because of its non-trivial validation pipeline.
5.3.1 Horse Status Management
A horse's status follows a state machine with five values: Available, Training, Unavailable, Paired, and Registered. Available is the default state for a newly created horse and is the only state from which pairing and registration are permitted. Training and Unavailable are owner-set states that temporarily remove the horse from the pairing pool: Training signals that the horse is being conditioned for a future race, while Unavailable signals injury, rest, or any other reason the horse should not be paired. The owner can freely switch a horse between Available, Training, and Unavailable at any time.
When an owner pairs a horse with a jockey, the system automatically transitions the horse's status to Paired; when that pair is registered for a race, the status automatically transitions to Registered. After the race is finalized by the Admin, the horse's status automatically returns to Available so that it can be entered in future races without manual intervention. Admins may override the status to Suspended in cases of misconduct or doping concerns, in which case the horse is removed from the pairing pool until the Admin lifts the suspension. All transitions are recorded in the audit log with timestamp and triggering user.

5.3.2 Horse Portrait Upload
Portrait upload enforces a multi-stage validation and processing pipeline to keep storage costs predictable and rendering fast. The client-side validation accepts only JPEG, PNG, and WebP files with a maximum size of 5 MB and a minimum resolution of 200×200 pixels; the maximum accepted resolution is 4096×4096, and larger images are auto-resized to those dimensions before upload. On the server side, the uploaded file is decoded, validated for true image content (not just extension), and resized to two derived assets: a 400×400 portrait used in horse cards and detail pages, and a 100×100 thumbnail used in compact lists. The original uploaded file is retained for archival purposes.
The three assets (original, portrait, thumbnail) are stored in S3-compatible object storage (MinIO in development, AWS S3 in production) and served through a CDN. The database stores only the canonical URL of each asset, never the binary content, which keeps the database small and avoids inflating backup windows. URLs are signed with a long expiry so that CDN caching is effective, and asset deletion (on horse archival) is performed asynchronously to avoid blocking the user request. Failed uploads are reported with a specific error code so that the client can show a meaningful message rather than a generic "upload failed".
5.4 Jockey Tab (Hiring)
The Jockey Tab lets the owner browse available jockeys and send pairing invitations. The tab lists jockeys whose status is Available (i.e., not currently under contract with another owner), with their career stats, win rate, and a thumbnail portrait. Filters allow narrowing by achievement badges (First Win, Five Races Completed, Sprint Specialist, Triple Crown) and by recent performance. Clicking a jockey opens a detail view with full career history and an Invite button that opens a short message dialog.
When the owner submits an invitation, the jockey receives an in-app notification (and an email if they have enabled email notifications) containing the owner's name, the horse's name, and the owner's message. The jockey can Accept or Decline; on Accept, the pairing contract is created, the horse's status transitions to Paired, and the jockey's status transitions to Paired; on Decline, the invitation is archived and the owner receives a notification. An invitation expires automatically after 72 hours if no action is taken, after which the owner must send a new invitation if they still wish to pair with that jockey. The owner may have at most one outstanding invitation per jockey at a time to prevent spamming.
5.5 Paired/Registered Management
The Paired/Registered Management area is a tabbed view that separates Active Pairings, Registered for Race, and History. The Active Pairings tab lists every pairing contract currently in force, with the jockey's name, the horse's name, the contract creation date, and (if applicable) the race the pair is registered for. The Registered for Race tab narrows the view to pairs that have been registered for an upcoming race, with the race name, scheduled start, and a countdown to the grace-period deadline for withdrawal.
An owner may withdraw a pair from a race within the grace period (default 5 days before the scheduled start; see Section 11.3) without Admin intervention. Withdrawing within the grace period dissolves the pairing contract, returns both the horse and the jockey to Available status, and frees the slot in the race's participant list. After the grace period, withdrawal requires Admin approval: the owner submits a withdrawal request, the Admin reviews and either approves (with the same effects as a regular withdrawal) or rejects (the pair remains registered). The History tab shows all past pairings and withdrawals with timestamps and outcomes so that the owner can audit their own activity.
5.6 Leaderboard and Prizes
The Leaderboard tab shows the owner their position among all Horse Owners on the platform, ranked by total prize money earned across finalized races. The leaderboard is filterable by time window (last 30 days, last 90 days, all-time) and by race type (Sprint, Mile, Medium, Long). Each row shows the owner's name, verified badge, total earnings in VND, number of races won, and number of races participated in. The owner's own row is pinned to the top of the list regardless of rank so that they can always see their own stats at a glance.
Prizes are distributed using the 60/30/10 split described in Section 13.2: the first-place pair earns 60% of the prize pool, the second earns 30%, and the third earns the remaining 10% (calculated as the residual to guarantee the total equals the prize pool exactly). The leaderboard updates in real time on Admin finalization: the moment an Admin confirms a race result, the prize amounts are credited to the relevant owners and the leaderboard re-sorts within seconds. A separate Prizes tab shows a paginated history of all prizes earned by the owner, with deep links to the originating race and a copy of the official race report.
5.7 Personal Profile
The Personal Profile area lets the owner view and edit their account information, including display name, contact phone, stable name, and avatar. Email changes require verification through a confirmation link sent to the new address before the change takes effect, so that a compromised session cannot silently take over the account. The profile page also hosts the password change form, which requires the current password to be entered alongside the new password and confirmation, so that a stolen JWT alone cannot be used to change the password.
On submitting a password change, the server verifies the current password against the stored BCrypt hash, validates the new password against the platform's password policy (minimum 8 characters, at least one letter and one digit, not in a small denylist of common passwords), updates the hash, and invalidates all outstanding refresh tokens for the account except the one used to make the change. A confirmation notification is sent to the user's email so that an attacker who somehow obtained the current password would be detected by the legitimate user. The full event is recorded in the audit log.
6. Jockey Business Flows
The Jockey is the second producer-side role: they receive invitations from Horse Owners, form pairing contracts, and ride in races. The Jockey experience is intentionally simpler than the Horse Owner experience because jockeys do not own horses and do not initiate race registrations — they only accept or decline invitations. Six sub-areas are described below.
6.1 Dashboard
The Jockey dashboard is the landing page after login and provides a snapshot of the jockey's current contract, upcoming rides, and career stats. The top of the dashboard shows a counts strip with the number of pending invitations, the number of active contracts, the number of upcoming rides, and the total number of races completed. Below the strip, a prominent card shows the jockey's current contract (if any) with the paired horse, the paired owner, and (if registered) the upcoming race with a countdown timer. A side rail lists pending invitations and the most recent notifications.
The dashboard surfaces two key invariants of the jockey experience: first, a jockey can be under at most one active pairing contract at a time (so the "current contract" card is unambiguous); second, a jockey can receive multiple simultaneous invitations but can accept at most one, after which all other pending invitations are auto-declined with a courtesy notification. These invariants are enforced both in the UI (the Accept button is disabled on all but the most recent invitation when a contract is active) and in the backend (the accept endpoint fails fast if a contract is already active).
6.2 Personal Profile
The Jockey Personal Profile exposes the jockey's career stats, achievement badges, and leaderboard rank, and is visible to Horse Owners browsing for jockeys to invite. The profile shows total races, wins, second-place finishes, third-place finishes, win rate, and a breakdown by race type. The achievement badges described in Section 6.6 are displayed prominently. A "Jockey Leaderboard Rank" field shows the jockey's current rank among all jockeys, ranked by total wins.
The visibility of the profile to Horse Owners is a deliberate design choice: a jockey cannot opt out of being browsed because the pairing market depends on Owners being able to evaluate jockeys. However, the jockey can choose which contact information (phone, email) is visible; the platform's invitation system does not require direct contact because all communication is mediated through in-app notifications. The profile can be edited by the jockey at any time, but changes to career stats are server-authoritative and derived from finalized race results, never user-editable.
6.3 Invitation System
The Invitation System is the jockey's primary entry point into a pairing contract. When an owner sends an invitation, the jockey receives a notification with the owner's name, the horse's name and portrait, and a short message. The jockey can Accept or Decline; both actions are recorded with a timestamp. Accepting creates a pairing contract that remains active until the pair's race is both completed and finalized by the Admin, after which the contract auto-dissolves and both the horse and the jockey return to Available status. The 1-to-1 invariant is strictly enforced: a jockey cannot accept a second invitation while a contract is active.
Declining an invitation is a soft action: it does not penalise the jockey and does not prevent the owner from sending a future invitation (though a 24-hour cooldown applies to prevent re-invitation spam). If the jockey takes no action, the invitation auto-expires after 72 hours, and the owner is notified. The invitation system also supports a courtesy "Not Interested" button that declines the invitation and silently prevents the same owner from sending another invitation for the same horse for 7 days, which gives the jockey a way to softly signal that a particular pairing is not a fit.
6.4 Horse View
The Horse View shows the jockey's currently paired horse (if any) and a history of past pairings. The current-horse card displays the horse's portrait, name, breed, age, owner, status, and any upcoming race the pair is registered for. The history list shows every previous pairing with the horse's name, owner, race (if any), result, and the date the contract was dissolved. Deep links from each history row lead to the relevant race detail page.
The Horse View serves both operational and reflective purposes: operationally, the current-horse card is the jockey's primary entry point into race-day preparation; reflectively, the history list lets the jockey review past performances and identify patterns (e.g., a horse that consistently places in the top three at Sprint distances). The view is read-only with respect to the horse's data — the jockey cannot edit the horse's profile, only the owner can — but the jockey can leave a private note on a previous pairing that is visible only to themselves.
6.5 Race View
The Race View lets the jockey browse upcoming, live, and past races they have participated in. The Upcoming tab shows races the jockey is registered for, with the race name, type, distance, scheduled start, and countdown timer. The Live tab shows any race currently in progress that the jockey is participating in, with a visual racetrack and the jockey's pair highlighted. The Past tab shows completed races with the jockey's finishing position and the official race report.
The Live tab is the most operationally important view for a jockey: it provides the real-time racetrack animation, the position of the jockey's pair, and the race timer. The jockey cannot influence the race outcome (the simulation is randomised server-side), so the Live tab is informational rather than interactive; however, it is the jockey's primary spectator experience of their own race. The tab also surfaces the referee's live race notes so that the jockey is aware of any incidents as they are documented.
6.6 Achievements
Achievements are badges awarded to jockeys for meeting specific career milestones. Four badges are defined in the MVP: First Win (awarded on the jockey's first race win), Five Races Completed (awarded when the jockey has completed five races regardless of outcome), Sprint Specialist (awarded when the jockey has won three Sprint races), and Triple Crown (awarded when the jockey has won races of three different types). The badges are displayed on the jockey's profile and in the invitation list so that owners can quickly identify experienced jockeys.
The Triple Crown badge is the most ambiguous of the four, because there are four race types (Sprint, Mile, Medium, Long) and the badge is defined as winning three different types. The team's working interpretation is any three of the four types, but this is flagged as a gap in Section 21 because a stricter reading (e.g., the traditional Sprint + Mile + Long combination) could be more meaningful. Achievements are evaluated automatically by a scheduled job that runs after every race finalization, so they are awarded within minutes of the qualifying result becoming official.
7. Race Referee Business Flows
The Race Referee is the platform's on-the-ground authority: they monitor live races, document incidents, disqualify participants, and submit the official race report that triggers the Admin finalization chain. Referee accounts are Admin-created only (see Section 4.5) because the role carries exceptional authority. The Referee experience is structured into four sub-areas below, followed by the mandatory structured race-notes format.
7.1 Pre-Race Management
Pre-Race Management is the referee's preparation area before the race begins. The referee sees the list of assigned races, the participant list for each, and the race configuration (type, distance, scheduled start, surface, prize pool). The referee can review each registered pair's horse portrait and jockey profile, verify that the registered pair matches the actual starters, and pre-populate notes about weather, track conditions, or anything else relevant to the race. The referee cannot start the race until the race's status has transitioned to Standby (i.e., the guess lock has engaged and the field is finalised for that moment).
7.1.1 Disqualification Form Requirements
Disqualifying a pair is a serious action that requires structured justification. The Disqualification Form captures the following fields: Pair ID (the pair being disqualified), Horse Name (auto-populated from Pair ID but editable for clarity), Jockey Name (auto-populated similarly), Reason (free-text, minimum 20 characters), Category (dropdown: Medical, Rule Violation, Equipment Failure, Administrative, Other), Severity (dropdown: Minor, Major, Critical), and Referee Notes (optional free-text for additional context). The form cannot be submitted until the referee types the literal string "CONFIRM" in a final confirmation field, which prevents accidental disqualifications and creates an audit trail showing deliberate intent.
Once submitted, the disqualification takes effect immediately: the pair is removed from the race's active participant list, the horse's status is updated to reflect the DQ, and a notification is sent to the pair's owner and jockey. The disqualification is recorded in the race report and in the audit log. The referee can rescind a disqualification only before the race is finalized; after finalization, any change requires Admin intervention through the rejection-and-revision loop described in Section 14.3.
7.2 Live Race Monitoring
Live Race Monitoring is the referee's primary in-race workspace. The view shows a countdown timer, a visual racetrack with each pair's current position, and a panel of quick-action buttons for common events: Stumble, Interference, False Start, and Injury Observed. Clicking a quick-action button opens the structured race-notes form pre-populated with the corresponding Category and Severity. The referee can also open the form manually to capture any other event. All notes are timestamped to the second and indexed by Race Time (e.g., "0:45 into race") so that the post-race report can reconstruct the race timeline.
Critically, the referee cannot unilaterally stop a race. The platform's design decision is that a race, once started, runs to completion unless an Admin intervenes; the referee's role is to document, not to halt. If a severe incident occurs (e.g., a horse injury), the referee marks the affected pair as DNF (Did Not Finish), which removes the pair from active position tracking and lists it separately in the results as "DNF - [Reason]". The simulation stops generating position updates for the DNF pair, and the race continues for the remaining pairs. The referee's notes form the basis of the official race report and may trigger Admin review after the race.
7.3 Race Confirmation
Race Confirmation is the gate between Completed and Report Ready. After the race timer reaches zero and provisional positions are recorded, the referee reviews the finishing order, the incident list, and any disqualified participants. The referee can re-order finishers if the simulation's provisional order conflicts with documented incidents (e.g., a pair that crossed first but was documented as interfering should be demoted). When the referee is satisfied, they tick a confirmation checkbox and "sign" the report by typing their name in a signature field; the system then transitions the race status from Completed to Report Ready and notifies the Admin that a report is awaiting review.
The signature field is not a cryptographic signature but a typed acknowledgement that creates an audit record: the typed name is compared against the referee's account name, and a mismatch is logged as a warning. The checkbox-and-signature pattern was chosen over a single "Submit" button because it forces the referee to perform two distinct deliberate actions, reducing the chance of an accidental submission. Once submitted, the report can only be revised through the Admin rejection-and-revision loop.
7.4 Race Report
The Race Report is the comprehensive post-race document that becomes the official record once the Admin confirms finalization. The report contains: the official results table (finishing position, pair, horse, jockey, finish time, points); highlights for 1st, 2nd, and 3rd place (auto-generated summary text); a penalties section listing any in-race penalties with timestamps; a DQ summary listing disqualified pairs with reasons and categories; and a registered-vs-actual-starters comparison that flags any pair that was registered but did not start (e.g., withdrawn after Standby with Admin approval).
The report is generated automatically from the in-race notes and the finishing order, but the referee can edit the highlight text and add a free-text summary before submitting. After Admin confirmation, the report is locked and becomes the official record; any subsequent correction requires the Admin to unlock the report, which is logged as a separate audit event. The report is publicly visible on the race detail page after finalization, so that spectators and owners can review the basis for the official result.

8. Spectator Business Flows
The Spectator is the consumer-side role: they browse races, place guesses on pairs, watch live races, and receive rewards based on the finishing position of their guessed pair. The Spectator experience is intentionally low-friction to maximise engagement: registration is immediate, guessing requires no payment, and rewards are in-kind rather than monetary. Six sub-areas are described below.
8.1 Personal Profile
The Spectator Personal Profile shows the user's account information, guess history, and reward history. The guess history lists every guess the spectator has placed, with the race name, the guessed pair, the finishing position of the pair (once the race is finalized), and the reward received (if any). The reward history lists every reward the spectator has earned, with the race, the position, the reward type, and a deep link to claim or view the reward (e.g., a voucher code or a horse-goods image and contact link).
The profile is read-only with respect to guess and reward history (these are server-authoritative) but editable with respect to display name, avatar, and notification preferences. The spectator can also export their guess history as a CSV file for personal record-keeping. Password changes follow the same flow as for Horse Owners (current password required) and are likewise logged in the audit trail.
8.2 Dashboard
The Spectator dashboard is the landing page after login and shows a snapshot of interesting races and the spectator's own activity. The top of the dashboard shows a counts strip with the number of guesses placed, the number of rewards earned, the number of upcoming races open for guessing, and the number of live races. Below the strip, a list of upcoming races open for guessing is displayed, each with a Place Guess button. A side rail lists the spectator's most recent guesses and notifications.
The dashboard is designed to make the path from login to placing a guess as short as possible: a spectator who logs in and finds an interesting race can place a guess in two clicks. Live races are surfaced with a Watch Live button so that spectators can join a race in progress without navigating to a separate page. The dashboard data is fetched through a single aggregated endpoint and cached for thirty seconds to balance freshness against load, with the cache busted immediately when the spectator places or modifies a guess.
8.3 Race Viewing
The Race Viewing area lets the spectator browse all visible races, filtered by status (Registration Open, Standby, In Progress, Completed), type (Sprint, Mile, Medium, Long), and distance range. Each race card shows the race name, type, distance, scheduled start, current participant count, prize pool, and the list of registered pairs. Clicking a card opens the race detail page, which shows the full pair list with horse portraits, jockey names, and (for completed races) finishing positions.
Race detail pages also surface the prize pool breakdown (60/30/10 split, displayed prominently) and the assigned referee. For completed races, the official race report is shown at the bottom of the page. The Race Viewing area is read-only with respect to race data — spectators cannot edit races — but it provides the deep links needed to place a guess (for open races) or watch live (for in-progress races).
8.4 Guess System
The Guess System is the spectator's primary engagement mechanic. Each spectator may place exactly one guess per race, selecting a single pair. The guess can be modified at any time until the race transitions to Standby (at which point guesses lock, as described in Section 12.2). The lock is enforced at the database level through a constraint that prevents updates to a guess whose associated race is in Standby or later, so that race conditions between the spectator's click and the status transition cannot bypass the lock.
The single-guess-per-spectator-per-race rule is enforced by a unique constraint on the (spectator_id, race_id) pair in the guesses table, so that even a bug in the application layer cannot create duplicate guesses. Multi-account circumvention is addressed declaratively through the Terms of Service (multi-accounting is prohibited and grounds for termination) rather than through technical fingerprinting, which the team judged out of scope for the MVP. No separate race registration is required for spectators: the guess record itself is the link between the spectator and the race.
8.4.1 Guess Placement Flow
Placing a guess follows a six-step flow designed to be both fast and explicit. First, the spectator navigates to a race detail page whose status is Registration Open. Second, they click the Place Guess button on the pair of their choice. Third, a confirmation dialog opens showing the pair, the race, and the prize-pool breakdown. Fourth, the spectator confirms; the guess is created in the database with status Active. Fifth, the UI updates to show the guess as placed, with a Modify Guess button visible until the race transitions to Standby. Sixth, when the race transitions to Standby, the guess is locked and the Modify Guess button is replaced by a Locked indicator.
Each step of the flow is idempotent: if the spectator double-clicks the Place Guess button, the second click is a no-op because the unique constraint rejects the duplicate. If the spectator navigates away and returns, the UI re-fetches the guess state and shows the current status. If the spectator attempts to place a guess on a race that has already transitioned to Standby, the backend rejects the request with a 409 Conflict and the UI shows a friendly message explaining that the guess window has closed.
8.5 Live Race Viewing
Live Race Viewing shows a visual racetrack animation of the race in progress, with each pair's current position updated at fixed intervals. The spectator's guessed pair (if any) is highlighted with a coloured border and a small badge so that the spectator can track their pair at a glance. A race timer shows elapsed time since the race start, and the referee's live race notes are surfaced in a side panel so that spectators are aware of incidents as they are documented.
The animation is driven by polling the /api/v1/races/{id}/positions endpoint at a configurable interval (default 1 second in production, 250 milliseconds in demo mode). WebSockets were considered but rejected for the MVP to keep the backend simpler; polling is sufficient because the simulation updates positions at intervals longer than the polling rate. The Live Race Viewing area is read-only — spectators cannot influence the race — and is the primary spectator experience of the racing action.
8.6 Guess Result Notification
Guess Result Notifications are sent only after the Admin finalizes the race. The notification informs the spectator of the finishing position of their guessed pair and, if applicable, the reward they have earned. Rewards are based on where the guessed pair finishes: 1st place earns horse goods (image, link, and contact), 2nd place earns a voucher code, 3rd place earns a drink coupon, and 4th or worse earns nothing. The notification is delivered through the in-app notification system and (if the spectator has enabled email notifications) via email.
If no spectator guessed the winning pair correctly (i.e., no spectator guessed the pair that finished 1st), no rewards are distributed and a "no correct guesser" message is displayed on the race detail page. This rule prevents the platform from manufacturing winners when none exist, which the team judged more honest than awarding the closest guess. The notification system is described in detail in Section 17, including the priority matrix that determines which channels are used for which notification types.
9. Admin Business Flows
The Admin is the platform's back-office operator: they verify accounts, create tournaments and races, assign referees, verify pairs, manage horses and jockeys, finalize race results, manage guesses and rewards, and cancel races when necessary. The Admin experience is the broadest of the five roles and is structured into nine sub-areas below.
9.1 Dashboard
The Admin dashboard is the landing page after login and provides a system-wide snapshot of operational health. The top of the dashboard shows a counts strip with the total number of users by role (with the number of pending Horse Owners and Jockeys broken out), the number of active tournaments and races, the number of pending race reports awaiting review, the number of guesses placed in the last 24 hours, and a system-health indicator that aggregates backend uptime, database latency, and email-service availability. Below the strip, a list of pending verifications (accounts and pairs) and a list of pending race reports are displayed with quick-action buttons.
The dashboard also surfaces system health flags such as "email service unreachable" or "audit log size above threshold", which the Admin can investigate immediately. The dashboard data is fetched through a single aggregated endpoint that pulls from multiple services; the response is cached for fifteen seconds to avoid hammering the database when multiple Admins are viewing the dashboard simultaneously. Each metric on the dashboard is clickable, leading to a detail page where the Admin can take the relevant action (e.g., the pending-reports count links to the report review queue).
9.2 Account/Role Management
Account/Role Management lets the Admin verify pending accounts, reassign roles, and create Referee accounts. Horse Owner and Jockey accounts require Admin verification before they can perform their role-specific actions; Spectator accounts are activated immediately and require no verification; Referee accounts are Admin-created only and cannot be created through the public registration form. The Admin can reassign roles (e.g., promote a Spectator to Jockey) with a confirmation dialog explaining the implications; the user's history is preserved and a notification is sent informing them of the change.
Verified accounts display a small check-mark badge next to the user's name throughout the platform, so that other users can distinguish verified users from unverified ones at a glance. The badge is purely cosmetic in the sense that Horse Owners and Jockeys can resume business without verification (they can complete their profile and even receive invitations), but they cannot perform the verification-gated action (register a horse, register a pair for a race) until verified. The Admin can also suspend or terminate an account in cases of misconduct, which is logged in the audit trail with the reason and the Admin's identity.
9.3 Tournament and Race Management
Tournament and Race Management provides CRUD operations for tournaments and races. The Admin can create a tournament (name, description, start date, end date), create races within a tournament, edit race details before the race enters Standby, and archive races after finalization. Race editing is locked once the race transitions to Standby so that the field is finalised for the guess lock; further changes require cancellation and re-creation. The Admin can also schedule races in batches through a CSV upload, which is useful for populating a tournament quickly during the demo.
9.3.1 Race Creation Form
The Race Creation Form captures the following fields: Race Name (free text), Tournament (dropdown of existing tournaments), Race Type (dropdown: Sprint 1000-1400m, Mile 1401-1800m, Medium 1801-2400m, Long 2401m+), Specific Distance (integer within the type's range, defaulting to the midpoint), Scheduled Date/Time (datetime picker, must be in the future), Max Participants (integer 6-18, default 12), Prize Pool (integer VND, can be 0), Surface (always Turf, displayed as read-only), and Registration Deadline (auto-calculated as one week before the scheduled start, displayed as read-only). If the Admin enters a prize pool of 0, a confirmation dialog appears: "This race has no prize pool. Is this intentional?"
The form validates the Specific Distance against the chosen Race Type's range so that an Admin cannot create a 1500m Sprint (which would be a Mile). The Scheduled Date/Time must be at least one week in the future in production (so that the auto-calculated registration deadline is in the future); in Demo Mode this constraint is relaxed to thirty minutes. The Max Participants value must be at least 6 (the minimum required for a race to be considered viable) and at most 18 (the maximum supported by the simulation). The form is submitted through a single transactional endpoint so that a failure rolls back all changes.
9.4 Pair Verification
Pair Verification is the Admin's review of registered pairs before the race begins. The Admin can review each registered pair's horse portrait, jockey profile, and registration timestamp, and can verify the pair (which leaves it in the race) or remove it (which unregisters it and notifies the owner and jockey). Verification can be performed individually or in bulk through a checkbox-and-verify-all action. The Admin can also remove a pair with a mandatory reason, which is recorded in the audit log and communicated to the affected parties.
Pair Verification is optional in the sense that a race can proceed without Admin verification if all pairs are auto-verified (the default); however, the Admin can enable mandatory verification on a per-tournament basis if they wish to scrutinise registrations more carefully. When mandatory verification is enabled, a race cannot transition to Standby until every registered pair has been verified or removed, which prevents last-minute surprise entries. The verification status of each pair is visible to the referee in the pre-race management view so that they know which pairs have been vetted.
9.5 Jockey and Horse Management
Jockey and Horse Management lets the Admin list all jockeys and horses on the platform with their status, owner (for horses), pairing history, and recent race results. The Admin can override a horse's status to Suspended in cases of misconduct or health concerns, which removes the horse from the pairing pool until the Admin lifts the suspension. The Admin can also override a jockey's status to Suspended, which prevents them from accepting new invitations; existing contracts remain in force until they naturally dissolve.
Status overrides are logged in the audit trail with the reason and the Admin's identity, and the affected owner or jockey is notified. The Admin can also edit non-authoritative fields such as the horse's name or breed (in case of typos at registration time), but authoritative fields such as race results are never editable through this interface. A search-and-filter panel allows the Admin to find specific horses or jockeys by name, owner, status, or recent activity, which is essential as the platform grows beyond a few hundred entities.
9.6 Referee Assignment
Referee Assignment lets the Admin assign a referee to each race. The system enforces one referee per race and prevents overlapping assignments on the same date so that no referee is double-booked. The Admin can reassign a referee before the race starts (i.e., before the race transitions to In Progress); after that, the assigned referee cannot be changed. The assignment is recorded in the audit trail, and the referee receives a notification immediately upon assignment.
If a referee becomes unavailable at short notice (e.g., illness), the Admin can reassign to another referee who has no conflicting assignment on that date. If no such referee is available, the Admin can either postpone the race (which requires cancellation and re-creation, because rescheduling is a separate optional function) or designate an Admin to act as referee for that race. The latter is a rare fallback that is logged prominently in the audit trail. In Demo Mode, the no-overlapping-assignments rule is relaxed to allow the same referee to oversee consecutive same-day races, as described in Section 20.
9.7 Race Result Finalization
Race Result Finalization is the Admin's review of the referee's submitted report. Once the referee submits the report (transitioning the race to Report Ready), the results are visible to all users with a "Provisional" banner until the Admin confirms. The Admin reviews the finishing order, the incident list, the DQ summary, and the registered-vs-actual-starters comparison. If satisfied, the Admin clicks Confirm, which transitions the race to Official: the leaderboard is updated, prizes are distributed according to the 60/30/10 split, and guess rewards are calculated and notifications are sent to spectators.
If the Admin is not satisfied, they can Reject the report, which loops back to the referee with specific revision notes. The race returns to a Revision Requested state, the referee updates the report and resubmits, and the review restarts. There is no limit on the number of revision cycles, which is flagged as a gap in Section 21 (the team's recommendation is to cap revisions at three and escalate after a 72-hour timeout). All confirmation and rejection actions are logged in the audit trail with timestamps and the Admin's identity.
9.8 Guess Management
Guess Management lets the Admin view guess counts per pair per race, confirm the 1st, 2nd, and 3rd place winners (cross-referenced with the finalized race results), and trigger reward distribution. Once the race is finalized and the official finishing order is determined, the system automatically identifies the spectators who guessed the winning pair and assigns rewards: 1st-place guessers receive a horse-goods reward (image, link, and contact), 2nd-place guessers receive a voucher code, and 3rd-place guessers receive a drink coupon. The Admin can review the reward assignments before they are sent, and can override a reward in exceptional cases (e.g., to substitute a different voucher if a specific one is out of stock).
Aggregate guess statistics are also surfaced in this view: total guesses per race, distribution of guesses across pairs, and the percentage of guesses that resulted in rewards. These statistics are useful for the Admin to identify popular pairs and to tune the reward mix over time. All reward assignments are logged in the audit trail and in the financial-operations category of the audit log, so that the platform has a complete record of every reward ever issued.
9.9 Race Cancellation
Race Cancellation is the Admin's nuclear option for a race that cannot proceed. The Admin can cancel a race at any point before the race starts (i.e., before the race transitions to In Progress) by providing a mandatory reason. Cancellation auto-unregisters all pairs (dissolving pairing contracts if the pair is not registered for any other race), voids all guesses placed on the race, and notifies all affected parties (owners, jockeys, spectators, referee). The race's status becomes Cancelled, and the race remains visible on the platform with the cancellation reason displayed.
An optional reschedule function allows the Admin to create a new race with the same configuration and re-invite the affected pairs to register; however, this is a manual process and the new race is treated as a separate entity. After cancellation, any audit log entries related to the race remain in place so that the cancellation is traceable. In Demo Mode, cancellation is rarely needed because races can be scheduled with short notice, but the function is preserved in production for genuine emergencies such as referee illness or system outage.

10.2 Race Simulation and Timer System
The race simulation is intentionally non-physical: it does not model horse stamina, jockey skill, or track conditions, and the finishing order is not deterministic. When the referee starts a race, the system generates a randomized total duration within the type's range (e.g., between 60 and 90 seconds for a Sprint). A countdown timer is displayed to all viewers, and position updates are generated at fixed intervals (every 5 seconds by default), with each pair's position shifted randomly within a bounded range so that the visualisation feels dynamic without becoming chaotic.
The ultimate finishing order is determined by a single randomisation event at the moment the timer reaches zero: the system assigns each remaining (non-DNF) pair a random finishing position, weighted by their position at the moment of timer expiry so that a pair leading near the end is more likely (but not guaranteed) to win. If the referee marks a horse as DNF during the race, that horse is immediately removed from position tracking and the simulation stops generating updates for it; the horse is listed separately in the results as "DNF - [Reason]". The simulation is server-side authoritative so that all viewers see the same positions, and the final order is recorded immutably the moment it is generated.
10.3 Race Status Lifecycle
A race progresses through eight lifecycle states, plus a Cancelled side-state that can be entered from any pre-start state. The lifecycle is: Draft (Admin is creating the race, not yet visible to non-Admins); Registration Open (the race is visible and pairs can register); Registration Closed (registration deadline has passed, no new pairs can register); Standby (the race is within the lock window, guesses are locked, the field is finalised for that moment); In Progress (the referee has started the race and the simulation is running); Completed (the timer has reached zero, positions are provisional); Report Ready (the referee has submitted the race report); and Official (the Admin has confirmed the report, results are final).
A critical design decision is that the Admin creates and configures races but does NOT start them: only the assigned Referee can transition a race from Standby to In Progress. This separation of concerns ensures that the Admin's configuration authority cannot be used to influence the race's outcome by selectively starting it at an advantageous moment. The Referee's start action is logged in the audit trail with a timestamp, and the simulation begins immediately. The Cancelled state can be entered by the Admin from any state up to and including Standby (i.e., before the race starts); once In Progress, a race cannot be cancelled but can be voided only through the post-race rejection-and-revision loop.

10.4 Demo Mode Time Compression
For the SWP391 demo, the natural lifecycle (which can span a week or more in production) is compressed into minutes through Demo Mode. When Demo Mode is enabled, races can be scheduled same-day, the one-week registration deadline is shortened to a configurable number of minutes, and the five-day grace period for pair withdrawals is shortened similarly. Once a race is finalized, the next race can open registration immediately, allowing the mentor to witness several complete lifecycles in a single demo session. All such compressions are surfaced in the UI via a "DEMO MODE" badge.
The documented exceptions to the standard rules in Demo Mode are: the registration deadline may be as short as 5 minutes; the grace period may be as short as 5 minutes; the pairing contract may dissolve immediately on finalization rather than after a cooling period; and the no-overlapping-referee-assignments rule may be relaxed to allow the same referee to oversee consecutive same-day races. These exceptions are clearly labelled in the UI as demo exceptions so that the mentor is not misled about the production behaviour. Demo Mode is implemented as a feature flag so that the production lifecycle can be restored simply by toggling the flag off.
11. Jockey-Horse Pairing System
11.1 Pairing Contract Model
The pairing contract is the temporal binding between a jockey and a horse for the purpose of racing in one or more races. A contract is created when a jockey accepts an owner's invitation, and it remains active until the pair's race is both completed and finalized by the Admin, after which the contract auto-dissolves and both the horse and the jockey return to Available status. While the contract is active, the jockey cannot accept other invitations and the horse cannot be paired with other jockeys (the 1-to-1 invariant). The contract is the unit of registration: a race entry references a contract, not a horse or a jockey individually.
The contract model was chosen over a more flexible many-to-many model (in which a jockey could be paired with multiple horses simultaneously) because it dramatically simplifies the validation logic and prevents a class of conflicts (e.g., a jockey accidentally riding two horses in the same race). The contract's lifecycle is automated: the system itself transitions the contract from Active to Dissolved at the moment of finalization, so no manual intervention is required. The contract record is permanent (it is never deleted, only transitioned) so that the full pairing history of every horse and jockey is queryable for the leaderboard and achievement calculations.
11.2 Pairing Rules and Constraints
Four hard rules govern the pairing system and are enforced at the database and service layers. First, the 1-to-1 rule: a jockey can be paired with at most one horse at a time, and a horse can be paired with at most one jockey at a time. Second, the Available-only rule: only horses in the Available status can be paired (a horse in Training, Unavailable, Paired, Registered, or Suspended cannot be paired). Third, the deadline rule: a pair must be formed (i.e., invitation accepted) before the race's registration deadline, because registration requires an active contract. Fourth, the permanence rule: a pairing record is permanent once created and is never deleted, even after dissolution, so that the audit trail and historical queries remain intact.
These four rules are enforced through a combination of database constraints (unique indexes on the active-contract view) and service-layer validation (pre-checks before creating a contract). The combination is deliberate: the database constraints are the ultimate guarantee against race conditions, while the service-layer validation produces friendly error messages that guide the user rather than surfacing raw SQL constraint violations. Any attempt to violate a rule is logged in the audit trail so that attempts to circumvent the system are traceable.
11.3 Grace Period and Withdrawal
The grace period for pair withdrawals is a hard rule: exactly 5 days (120 hours) before the scheduled race start time, after which withdrawal requires Admin approval. The grace period is auto-calculated from the race's scheduled start time and displayed to the owner during registration and on the pair management page, so that the owner always knows how long they have to withdraw without intervention. The grace period is configurable per tournament at 3, 5, or 7 days, with 5 days as the system default; the chosen value is recorded against the tournament and applied to every race within it.
After the grace period expires, withdrawal is still possible but requires the owner to submit a withdrawal request that the Admin must approve. The Admin reviews the request with the reason provided and either approves (with the same effects as a regular withdrawal: contract dissolved, horse and jockey returned to Available, race slot freed) or rejects (the pair remains registered). The Admin's decision is logged in the audit trail and communicated to the owner. In Demo Mode, the grace period may be shortened to minutes so that the mentor can witness both the in-grace-period and post-grace-period withdrawal flows within a single session.
12. Spectator Guess System
12.1 Betting Rules
The spectator guess system is governed by a small set of betting rules that keep the mechanic simple and fair. Each spectator may place exactly one guess per race, selecting a single pair from the registered field. The guess can be modified at any time until the race transitions to Standby, at which point the guess is locked and cannot be modified or deleted. Multi-account circumvention of the one-guess-per-race rule is addressed declaratively through the Terms of Service (prohibited and grounds for termination) rather than through technical fingerprinting, which the team judged out of scope for the MVP.
No separate race registration is required for spectators: the guess record itself is the link between the spectator and the race. This design choice eliminates an entire class of registration management (capacity, deadlines, withdrawals) that would otherwise be duplicated for spectators. The guess record is lightweight (just spectator_id, race_id, pair_id, status, and timestamps) and is stored in a dedicated guesses table with a unique constraint on (spectator_id, race_id) so that duplicate guesses are impossible even if the application layer has a bug.
12.2 Lock Mechanism
The lock mechanism is triggered automatically when the race transitions to Standby, which happens at a configurable time before the scheduled race start (default 5 minutes in production, 1 minute in Demo Mode). At the moment of transition, every guess associated with the race is set to Locked status in a single transaction, and a database-level constraint prevents any further updates to a guess whose associated race is in Standby or later. The constraint is the ultimate guarantee: even if the application layer attempts an update (due to a bug or a malicious request), the database rejects it with a constraint violation.
The lock is visible to spectators as a Locked indicator on their guess, replacing the Modify Guess button that was visible before Standby. The indicator is non-interactive: once locked, a guess cannot be unlocked by the spectator under any circumstances, and the only way to change the guessed pair would be to cancel the race (which voids all guesses) and create a new one. The lock is essential for the integrity of the reward distribution: if guesses could be modified after the race started, a spectator could wait for the leading pair to become clear and then "guess" on it retroactively, which would defeat the purpose of the system.
12.3 Reward Distribution
Reward distribution occurs only after the Admin confirms the race result (i.e., the race transitions to Official). The system identifies all spectators whose guessed pair finished in 1st, 2nd, or 3rd place (in the official, post-revision finishing order) and assigns rewards: 1st-place guessers receive a horse-goods reward (image, link, and contact), 2nd-place guessers receive a voucher code, 3rd-place guessers receive a drink coupon, and 4th-or-worse guessers receive nothing. All rewards are delivered through in-app notifications, and (if the spectator has enabled email notifications) via email.
If no spectator guessed the winning pair correctly (i.e., no spectator guessed the pair that finished 1st), no rewards are distributed and a "no correct guesser" message is displayed on the race detail page. This rule is more honest than awarding the closest guess, because it preserves the integrity of the guess as a prediction rather than a proximity. The rule also means that the platform's reward budget is contingent on actual correct guesses, which keeps the cost of the demo predictable: if the mentor's pair loses, no rewards are issued, and the platform's finances are unaffected.

13. Prize Pool System
13.1 Display-Only Model
The prize pool is a display-only field set by the Admin when creating a race. It is a fixed value denominated in VND (Vietnamese Dong), and it does not change based on participant count or guesses. The prize pool can be 0, in which case the race is displayed with an "Exhibition Race — No Prize Pool" badge and the prize section is hidden from the race detail page. The prize pool is not connected to any real financial transaction: it is a virtual number displayed for the engagement value of having a tangible stakes figure, and the actual rewards to spectators are in-kind goods as described in Section 12.4.
The display-only model was chosen to avoid the regulatory complexity of handling real money, which would require payment gateway integration, KYC procedures, anti-money-laundering controls, and tax reporting — all of which are far out of scope for a university capstone project. By keeping the prize pool purely virtual and pairing it with in-kind spectator rewards, the platform preserves the excitement of racing for tangible stakes without exposing itself to financial regulation. The VND denomination was chosen for local relevance and because VND has no sub-unit (no cents), which simplifies the distribution mathematics described in Section 13.2.
13.2 Distribution Mathematics
Prize distribution follows the 60/30/10 split: the first-place pair earns 60% of the prize pool, the second earns 30%, and the third earns the residual 10%. The residual is calculated by subtraction rather than as 10% of the prize pool, to guarantee that the three shares sum exactly to the prize pool even when rounding introduces a small discrepancy. The exact formula is:
First  = Math.round(PrizePool * 0.60)
Second = Math.round(PrizePool * 0.30)
Third  = PrizePool - First - Second
Because VND has no sub-unit, Math.round() always produces an integer, and the subtraction guarantees that First + Second + Third == PrizePool. For example, a prize pool of 1,000,000 VND yields First = 600,000, Second = 300,000, Third = 100,000 (sum 1,000,000). A prize pool of 1,000,001 VND yields First = 600,000, Second = 300,000, Third = 100,001 (sum 1,000,001). The residual-assignment-to-First approach favours the winner in cases of rounding discrepancy, which the team judged acceptable given that the prize pool is display-only and the discrepancies are at most 1 VND.
13.3 Tie/Dead-Heat Handling
Ties (dead heats) are handled by merging the tied positions and sharing the corresponding prize equally. For example, if two horses tie for 1st place, they share the 1st-place and 2nd-place prizes equally: each receives (60% + 30%) / 2 = 45% of the prize pool, and the next horse (which would otherwise be 3rd) becomes the de facto 3rd-place finisher and receives the 10% prize. Both tied horses' guessers win the full reward for their position (i.e., both receive horse-goods rewards), and both tied horses receive full points toward the leaderboard.
To minimise the frequency of ties, the simulation adds a random millisecond offset to each pair's finish time, so that exact ties are extremely unlikely (the probability of two random millisecond offsets matching is approximately 1 in 1000 per pair comparison). When a tie does occur despite this offset (which can happen if the rounding of the displayed finish time obscures the millisecond offset), the merged-positions-share-equally rule applies. The team considered instead using the millisecond offset as a tiebreaker (so that one horse is declared the winner and receives the full 60%), but judged that the merged-positions rule is more consistent with the spirit of the platform and more rewarding for spectators.
13.4 Zero Prize Pool
When the prize pool is 0, the race is displayed with an "Exhibition Race — No Prize Pool" badge in place of the prize section on the race detail page. The prize distribution logic is skipped entirely (no calculations are performed, no leaderboard updates occur for prizes, no notifications are sent about prize amounts). The race still affects the leaderboard for points (win, place, show counts) but not for prize money, so that exhibition races do not inflate any owner's prize total.
When the Admin enters a prize pool of 0 in the Race Creation Form, a confirmation dialog appears: "This race has no prize pool. Is this intentional?" This dialog prevents accidental 0-value prize pools from being created (e.g., because the Admin forgot to fill in the field). Once confirmed, the 0 prize pool is stored and the race is treated as an exhibition race for the rest of its lifecycle. The Admin can edit the prize pool later (before Standby) if they change their mind, with the same confirmation dialog appearing if the new value is 0.
14. Race Result Finalization Chain
14.1 End-to-End Flow
The finalization chain begins when the race timer reaches zero and provisional positions are recorded. At this moment the race transitions to Completed, the provisional finishing order is displayed to all viewers with a "Provisional" banner, and the referee is prompted to review and submit the race report. The referee reviews the finishing order, the in-race incident notes, and the disqualified participants, edits the report's highlight text if desired, and submits the report (transitioning the race to Report Ready). The Admin is notified that a report is awaiting review.
The Admin reviews the report: if satisfied, they click Confirm, which transitions the race to Official. At this moment the leaderboard is updated (prize money credited to the relevant owners, win/place/show counts incremented for the relevant jockeys and horses), guess rewards are calculated and assigned to the relevant spectators, and notifications are sent to all affected parties. The Official result is locked and becomes the permanent record; any subsequent correction requires the Admin to unlock the report, which is a separate audit event. If the Admin is not satisfied, they Reject the report, which loops back to the referee as described in Section 14.3.

14.3 Admin Rejection and Loop-Back
When the Admin rejects a referee report, the report goes back to the referee with specific revision notes provided by the Admin. The race transitions to Revision Requested, the referee is notified, and the report becomes editable again. The referee updates the report (e.g., re-orders finishers, adds an incident note, removes a disqualification) and resubmits, which transitions the race back to Report Submitted and restarts the Admin review. There is no limit on the number of revision cycles in the MVP, which is flagged as a gap in Section 21.
All rejection and resubmission actions are logged in the audit trail with timestamps, the Admin's or referee's identity, and the revision notes. This creates a complete audit trail of the finalization negotiation, which is essential if the result is later disputed by an owner or jockey. The team's recommendation (in Section 21) is to cap revisions at three cycles and to auto-escalate after a 72-hour timeout, with the escalation path being a senior Admin or the platform's product owner making a binding decision to break the deadlock.
14.4 Mid-Race Violation Handling
Mid-race violations are handled through a simplified model: violations are noted by the referee during the race using the structured race-notes form (Section 7), but the final judgement proceeds only after the race ends. The referee does not unilaterally disqualify a horse mid-race; instead, the violation is documented and the horse continues to be tracked. After the race ends, the referee processes the violations formally through the post-race report, which may include disqualifying the offending horse (removing it from the official order and listing it as "Disqualified - [Reason]").
This simplified model was chosen over a more reactive model (in which the referee could disqualify a horse mid-race and immediately remove it from tracking) because it preserves the integrity of the race simulation: a mid-race disqualification would require the simulation to re-randomise the finishing order for the remaining horses, which would be visible to spectators and could undermine confidence in the result. By deferring the disqualification to the post-race report, the platform ensures that the simulation runs to completion and that the disqualification is applied to a stable finishing order, which is easier to reason about and to audit.

16. Tournament Structure Decision
A full horse-racing tournament spans weeks or months, with multiple races accumulating points toward a final standings table and a tournament champion. The team evaluated this model carefully and decided that it is too far out of scope for the SWP391 MVP: a multi-week tournament cannot be demonstrated in a single mentor session, the aggregate-scoring logic adds significant complexity to the leaderboard, and the operational overhead of running multiple concurrent races exceeds what the team can support during the demo period. The decision was therefore made to simplify the tournament structure to its essential core.
In the MVP, each race is treated as the "final race" of a hypothetical prior tournament. This framing preserves the narrative stakes of a final (the race matters, the result is consequential, the rewards are tangible) without requiring the platform to model the prior tournament's accumulated standings. The Tournament entity still exists in the data model and can be created by the Admin, but it functions primarily as a grouping label: races are created within a tournament for organisational purposes, but no aggregate scoring is computed across the races within a tournament.
The implications of this simplification are: no aggregate scoring across multiple races in the MVP; the leaderboard reflects single-race results or all-time aggregated results (a separate query that does not depend on tournament membership); and the tournament entity is essentially a folder for races. Future iterations of the platform could add a full tournament system with point accumulation, multi-race champion calculation, and tournament-scoped leaderboards, but these are explicitly out of scope for v4.0 and are noted as future work in Section 22.
17. Notification System
17.1 Channels
The notification system uses three channels, each with a distinct role. The primary channel is in-app notifications, which are persisted in a notifications table and displayed in a bell-icon dropdown accessible from any authenticated page; in-app notifications are the default channel for all notification types and remain available until the user explicitly dismisses them. The secondary channel is email, used for critical events that the user must not miss (e.g., race cancellation, account rejection, password reset link); email is sent through a transactional email service and is always sent in addition to (never instead of) the in-app notification.
The tertiary channel is push notifications, which are optional and require the user to grant browser permission. Push is used for important real-time events (e.g., race starting soon, invitation received, guess result) when the user is not actively viewing the platform. Push is never used as the sole channel for any notification type, because push delivery is not guaranteed (the user may have denied permission, or the browser may be closed). All three channels are governed by the user's notification preferences, which can be tuned per channel in the personal profile.

18. Audit Log System
18.1 Scope
The audit log captures every state-changing action on the platform across five categories. First, authentication events: login, logout, failed login, password change, password reset, role reassignment. Second, CRUD operations on core entities: create, update, delete (or archive) for User, Horse, JockeyProfile, Tournament, Race, RaceEntry, PairingContract, Guess, RefereeAssignment, RaceReport, Notification, Prize. Third, status transitions: every change to a race's lifecycle state, a horse's status, a jockey's status, a pairing contract's status, or a guess's status. Fourth, admin actions: account verification, role reassignment, pair verification, race cancellation, report confirmation, report rejection, reward override. Fifth, financial operations: prize distribution, reward assignment, voucher code consumption.
The scope was chosen to be comprehensive enough to support post-incident investigations and compliance audits, but narrow enough to keep the audit log's growth rate manageable. Read-only queries (e.g., viewing a race detail page) are not logged in the audit trail because they do not change state; they are instead logged in the application's access log, which has a shorter retention period. The boundary between the audit log and the access log is enforced by an aspect-oriented interceptor that wraps every service method annotated with @Audited.
18.2 Storage Architecture
The audit log is stored in a dedicated table within the same MySQL database as the operational data, but with strict append-only enforcement: the database role used by the application has INSERT and SELECT privileges on the audit table but not UPDATE or DELETE, so that even a compromised application cannot rewrite history. The table is indexed by timestamp (for date-range queries), user_id (for per-user queries), and entity_type (for per-entity-type queries), so that the most common query patterns are supported without a full table scan.
Partitioning by month is planned for production to keep individual partitions small and to allow old partitions to be archived to cold storage. For the MVP, the table is unpartitioned and the team relies on the indexes to keep queries performant; retention policy is addressed as a gap in Section 21. The audit log is never exposed through a public API: it is accessible only through the Admin's query interface described in Section 18.4, and even the Admin can only read (not modify) the log.

18.4 Query Interface
The audit log query interface is accessible only to Admins and provides filters by date range, user, entity type, and action. The interface is read-only: no query modifies the log. Results are paginated (default 50 per page) and can be exported to CSV for compliance purposes (e.g., to share with a regulator or an external auditor). The export is rate-limited to one per minute per Admin to prevent abuse, and every export is itself logged in the audit trail so that the act of exporting is auditable.
The query interface is designed for investigative use rather than real-time monitoring: an Admin who needs to know "what happened to race #1234" can filter by entity_type = Race and entity_id = 1234 to see every audit event related to that race, in chronological order. For real-time monitoring of system health, the Admin dashboard (Section 9.1) surfaces aggregated metrics rather than raw audit events, so that the Admin is not overwhelmed by the volume of audit data during normal operations.
19. Password Reset & Account Recovery
19.1 Flow
The password reset flow begins when a user clicks the "Forgot Password?" link on the Login page. The user is taken to a form that asks for their email address; on submission, the system checks whether an account with that email exists and, if so, generates a single-use reset token and sends a reset link to the email. The link contains the token as a query parameter and is valid for 15 to 30 minutes (the exact validity is a configurable system setting, defaulting to 30 minutes). The user clicks the link, is taken to a reset form that asks for a new password and confirmation, and on submission the password is updated and the token is invalidated.
Until the reset link is used, the old password remains active, so a user who remembers their credentials after requesting a reset can simply log in normally and ignore the reset email. After the reset is used, a confirmation email is sent to the same address so that the user is aware that the password has been changed (and can take action if the change was not authorised by them). All reset requests, successful and failed, are logged in the audit trail under the authentication-events category.
19.2 Security
The reset link is single-use: the moment a new password is set, the token is invalidated and any subsequent attempt to use the link results in an error. The reset-request endpoint returns a generic success message regardless of whether the email exists in the system, to prevent user-enumeration attacks in which an attacker probes the system to discover which emails are registered. Rate limiting is applied to reset requests: a maximum of 3 requests per email per hour and a maximum of 10 requests per IP per hour, which prevents an attacker from spamming reset emails.
The reset token is a cryptographically random string of at least 32 bytes, generated using a secure random number generator and stored in the database as a BCrypt hash (so that a database leak does not expose valid tokens). The token's expiry is enforced server-side at validation time, so even a stolen token cannot be used after the expiry. The new password is validated against the platform's password policy (minimum 8 characters, at least one letter and one digit, not in a small denylist of common passwords) and rejected if it does not meet the policy.
19.3 Admin Override
In exceptional cases (e.g., a user has lost access to their email and cannot use the self-service reset flow), the Admin can force-reset a user's password directly. The Admin enters a new password on the user's behalf, the password is updated, all outstanding refresh tokens for the user are invalidated, and a confirmation email is sent to the user's registered email address informing them of the change. The Admin override is logged in the audit trail with the Admin's identity and the reason (which the Admin must provide).
The Admin override is intentionally friction-heavy: the Admin must navigate to the user's profile, click a "Force Reset Password" button, enter and confirm the new password, and provide a reason in a free-text field. This friction is deliberate to discourage casual use of the override and to ensure that it is reserved for genuine emergencies. The override does not bypass the password policy: the new password must still meet the same requirements as a self-service reset, and the user is encouraged to change it again at their earliest convenience through the normal profile flow.
20. Demo Mode Considerations
20.1 Why Demo Mode Exists
Demo Mode exists because the SWP391 demo cannot wait days or weeks for the natural race lifecycle to unfold. A mentor evaluation session lasts at most a few hours, during which the team must demonstrate the full lifecycle of the platform: account registration and verification, horse and jockey management, pairing, race registration, guessing, live racing, finalization, and reward distribution. With the production lifecycle (one-week registration deadline, five-day grace period, race scheduling days in advance), a single lifecycle would span two weeks, which is obviously incompatible with a demo session.
Demo Mode compresses the lifecycle without breaking the documented business rules: the rules are still enforced, but the time windows are shortened so that the mentor can witness a complete lifecycle in minutes. The compression is implemented as a feature flag so that the production lifecycle can be restored simply by toggling the flag off, and the demo and production code paths are identical except for the configurable time windows. This means any bug discovered during the demo is guaranteed to be reproducible in production, which is essential for the team's confidence in the production deployment.
20.2 Time Compression Rules
When Demo Mode is enabled, the following time windows are compressed. First, races can be scheduled same-day, with the start time as little as 30 minutes in the future (compared to one week in production). Second, the registration deadline is shortened from one week to a configurable number of minutes (default 5 minutes). Third, the five-day grace period for pair withdrawals is shortened to a configurable number of minutes (default 5 minutes). Fourth, once a race is finalized, the next race can open registration immediately, allowing consecutive races within a single session. Fifth, the no-overlapping-referee-assignments rule is relaxed to allow the same referee to oversee consecutive same-day races.
All such compressions are surfaced in the UI via a prominent "DEMO MODE" badge so that the mentor is not misled about the realism of the configuration. The badge appears in the header on every page when Demo Mode is active, and the compressed time windows are explicitly labelled as demo exceptions in the relevant UI elements (e.g., the registration deadline is displayed as "5 minutes (DEMO)" rather than just "5 minutes"). The team considered making the badge more subtle (e.g., only visible to logged-in users) but decided that a prominent badge is more honest and more useful for the mentor's evaluation.
20.3 Demo Flow Walkthrough
A typical demo session follows this sequence. The Admin creates a race with a start time of T+30 minutes (T being the current time). Over the next 5 minutes, Horse Owners register pairs (the Admin has pre-verified several owner and jockey accounts for the demo). Over the following 10 minutes, Spectators place guesses on their chosen pairs. At T-1 minute, the race transitions to Standby and guesses are locked. At T, the assigned Referee clicks Start, the race enters In Progress, and the simulation runs for approximately 1 to 3 minutes (depending on race type).
When the timer reaches zero, the race transitions to Completed with provisional positions. The Referee reviews the in-race notes, edits the report's highlight text if desired, and submits the report. The Admin reviews the report and clicks Confirm, transitioning the race to Official: the leaderboard is updated, prizes are distributed, guess rewards are calculated, and notifications are sent. The next race can now open registration immediately, and the cycle repeats. A complete lifecycle takes approximately 10 to 15 minutes, allowing the mentor to witness 3 to 4 lifecycles in a single one-hour demo session.
20.4 Documented Exceptions
The business rules whose enforcement is relaxed in Demo Mode are explicitly documented here so that there is no ambiguity about what is and is not permitted. The grace period may be shortened to minutes for demo (rule R11 is relaxed). The pairing contract may dissolve immediately on finalization rather than after a cooling period (rule R10 is relaxed). The no-overlapping-referee-assignments rule may be relaxed to allow the same referee to oversee consecutive same-day races (rule R14 is relaxed). All other rules remain in force: the 1-to-1 pairing invariant (R07, R08), the Available-only pairing rule (R09), the guess lock at Standby (R02), and so on.
These exceptions are clearly labelled in the UI as demo exceptions: when a compressed time window is displayed, the label includes the word "DEMO" so that the mentor is aware that the displayed value is a demo compression rather than the production default. The exceptions are also logged in the audit trail with the DEMO_MODE flag set, so that audit events generated during a demo session can be filtered out of production analytics. The team's intention is that Demo Mode is disabled in the production deployment, so that the relaxed rules never apply to real users; the feature flag is the single point of control for this.
21. Criticism, Gaps & Further Suggestions
This section is the author's original critical analysis of the document above. Even after resolving the 25 gaps inherited from v3 and adding the nine new systems in v4.0, the platform's business logic still contains unhandled scenarios, logical inconsistencies, and edge cases that the team should address before the production deployment. Each item below describes the gap, explains its impact, and proposes a concrete resolution. The 13 items are presented in approximate order of severity.
21.1 Triple Crown Achievement Ambiguity
The Triple Crown achievement is defined as winning races of three different types, but there are four race types (Sprint, Mile, Medium, Long). The current specification does not clarify whether Triple Crown means any three of the four, all four, or a specific traditional combination. This ambiguity will lead to inconsistent badge awarding depending on how the implementing developer interprets the rule, and jockeys may be surprised when a badge they expected is not awarded (or is awarded earlier than expected). Resolution: define Triple Crown explicitly as winning any three of the four race types, and rename the badge to "Three-Type Champion" to avoid confusion with the traditional horse-racing Triple Crown (which is a specific three-race series). Add a fourth badge, "Grand Slam", for winning all four types, to preserve the aspirational tier above Triple Crown.
21.2 No Revision-Cycle Limit on Finalization
Section 14.3 specifies that the Admin can reject a referee report and loop back to the referee indefinitely. This creates a deadlock risk: a stubborn referee and a stubborn Admin could loop forever, with the race stuck in Revision Requested and prizes never distributed. Spectators who guessed correctly would be denied their rewards indefinitely, and the platform's reputation would suffer. Resolution: cap revisions at three cycles; on the fourth rejection, the race enters a Deadlock-Escalated state and a senior Admin (or the product owner) is notified to make a binding decision within 72 hours. If no decision is made within 72 hours, the report is auto-confirmed as Official with a "timeout-confirmed" flag, and the incident is logged for post-mortem review.
21.3 Multiple Horses from the Same Owner in the Same Race
The current rules do not address whether a Horse Owner can register two or more horses in the same race. Allowing this creates a conflict of interest: the owner could place a guess on their own second-string horse to divert attention from their primary entry, or could collude between their own jockeys to influence the result. Even without malicious intent, multiple entries by one owner distort the guess distribution and the prize pool. Resolution: restrict to one horse per owner per race for the MVP, enforced by a service-layer check on race registration. Document this as a deliberate simplification; if the production deployment ever requires multiple entries, the rule can be relaxed with additional anti-collusion measures (e.g., forbidding the owner from guessing on any of their own horses).
21.4 Referee Account Creation vs. User Reassignment
Section 4.5 states that Referee accounts are Admin-created only, but does not specify whether an existing user (e.g., a Spectator) can be promoted to Referee or whether a brand-new account must always be created. If promotion is allowed, the user's spectator history (guesses, rewards) must be preserved or archived; if not, the platform loses potentially qualified referees who are already familiar with the system. Resolution: allow Admin to promote an existing user to Referee, preserving their spectator history in a read-only archive. The user gains the Referee role in addition to (not instead of) their existing role, so they can act as a referee in some races and as a spectator in others. The dual-role capability is gated by the same Admin verification flow as a new Referee account.
21.5 Spectator Guess on a Disqualified Pair
If a spectator guesses on a pair that is disqualified before the race starts (e.g., the pair is removed by the Admin during pair verification), the current rules do not address whether the spectator can re-guess on another pair. If they cannot, they are effectively locked out of the race through no fault of their own, which is frustrating and unfair. Resolution: when a pair is removed before Standby, all guesses on that pair are auto-voided and the affected spectators are notified with an invitation to re-guess on a remaining pair before the guess lock. If the pair is removed after Standby (which requires Admin override), the guesses remain locked and are treated as if the pair finished last (i.e., no reward); the affected spectators are notified and offered a courtesy voucher as compensation.
21.6 Race with All DNFs
The pathological case where every horse in a race is marked DNF (e.g., a mass injury or a catastrophic weather event) is not addressed. Without a winner, the 60/30/10 prize distribution cannot be applied, the leaderboard cannot be updated, and guess rewards cannot be calculated. Resolution: if all horses in a race are marked DNF, the race is automatically flagged as Void and the Admin is notified to confirm the void. On confirmation, the race transitions to Void status, all prizes and rewards are cancelled, all guesses are voided, and the affected owners and spectators are notified with an explanation. The race remains visible with a "Void - All DNF" badge and the referee's report is preserved for review.
21.7 Currency Precision and VND-Only Constraint
Section 13.2 uses Math.round() in the 60/30/10 split, which works for VND (no sub-unit) but would produce inconsistent totals for currencies with cents (e.g., USD with $1.00 = 100 cents). The document does not explicitly state that the prize pool is always VND, which leaves room for a future maintainer to introduce a different currency and encounter subtle rounding bugs. Resolution: add an explicit rule that the prize pool is denominated in VND only, and that the residual-to-Third formula in Section 13.2 assumes integer arithmetic. If a future iteration needs to support other currencies, the formula must be revisited to handle sub-units correctly (e.g., by working in the smallest sub-unit).
21.8 Leaderboard Freshness During Demo
Section 5.6 states that the leaderboard updates "real-time on Admin finalization," but in Demo Mode with rapid consecutive races, the UI may struggle to keep up if the leaderboard is fetched via polling. A spectator viewing the leaderboard during a demo session might see stale data for several seconds after a finalization, which undermines the impact of the demo. Resolution: implement a lightweight event bus (Spring Application Events) that emits a LeaderboardUpdated event on finalization, and have the leaderboard page subscribe via Server-Sent Events (SSE) so that the UI refreshes within milliseconds of the finalization. SSE is preferred over WebSockets for this use case because the data flow is one-way (server to client) and SSE has simpler reconnection semantics.
21.9 Notification Spam During Demo
Rapid demo races could spam spectators with notifications (guess locked, guess result, race starting, reward received) at a rate that overwhelms the bell-icon dropdown and the email inbox. This is especially problematic if the demo runs four or five races in an hour, each generating five or more notifications per spectator. Resolution: when Demo Mode is active, apply notification throttling: Informational notifications are batched into a single digest per race (rather than sent individually), and email is suppressed for Informational and Important priorities (only Critical emails are sent in Demo Mode). The throttling is reverted automatically when Demo Mode is disabled, and the throttle settings are documented in the notification system's configuration so that they can be tuned per demo.
21.10 Audit Log Growth and Retention Policy
The audit log is append-only and never deleted, which means it grows unbounded with platform usage. In a production deployment with hundreds of races per month, the audit table could grow to billions of rows within a year, degrading query performance and inflating backup windows. The current specification does not address retention. Resolution: define a retention policy of 90 days hot (queryable via the Admin interface) and 7 years cold (archived to object storage in Parquet format, queryable on demand via a separate export-and-query workflow). A scheduled job moves rows older than 90 days from the hot table to the cold archive nightly, and the Admin interface displays the date range of the hot data so that the Admin knows when to escalate to a cold-archive query.
21.11 Email Service Dependency and Fallback
Password reset and critical notifications depend on the email service. If the email service is down, password resets cannot be delivered (locking users out of their accounts), race cancellation emails cannot be sent (leaving affected parties uninformed), and account rejection emails cannot be sent (leaving pending users in limbo). Resolution: implement a retry queue with exponential backoff for all email sends (3 retries over 15 minutes), and surface email-service unavailability on the Admin dashboard as a system-health flag. For password resets specifically, provide a fallback: if the email service is down, the Admin can issue a reset token directly through the Admin interface and communicate it to the user out-of-band (e.g., via phone), with the action logged in the audit trail. The fallback is intentionally friction-heavy to discourage casual use.
21.12 Referee Concurrent Race Assignment in Demo Mode
Section 9.6 enforces no overlapping referee assignments on the same date, which is correct for production but blocks the demo scenario where the same referee oversees consecutive same-day races. Section 20 mentions that Demo Mode relaxes this rule, but does not specify the relaxation's bounds (e.g., minimum gap between races, maximum races per referee per day). Without bounds, a referee could be assigned to two races whose windows overlap, creating a genuine conflict. Resolution: in Demo Mode, allow the same referee to oversee multiple same-day races provided no two races' In Progress windows overlap (i.e., the second race's start time must be after the first race's expected end time plus a 5-minute buffer). The assignment UI enforces this constraint and shows the referee's existing same-day assignments as a visual timeline so the Admin can see potential conflicts at a glance.
21.13 Pairing Contract Edge Case on Post-Grace Withdrawal
If an owner withdraws a pair after the grace period with Admin approval (per Section 11.3), the current rules do not clearly specify whether the jockey's 1-to-1 contract dissolves immediately or remains active until the would-have-been race is finalized. If the contract remains active, the jockey is locked into a contract for a race that will never happen, which is unfair to the jockey and blocks them from accepting other invitations. Resolution: stipulate that an approved post-grace withdrawal dissolves the pairing contract immediately, returning both the horse and the jockey to Available status. The contract record is preserved in the audit trail with a Withdrawn-Approved status, and both parties are notified. This rule should be added to Section 11 as an explicit clarification of rule R12.
These 13 items are not exhaustive: a longer review would likely surface additional edge cases in areas such as internationalisation (the platform is in English but the course is in Vietnam — should Vietnamese be supported?), database migration on rule changes (if the grace period default changes from 5 to 7 days, how are existing races handled?), and race-result dispute resolution (after Admin finalizes, can a Horse Owner appeal?). The team should treat Section 21 as a starting point for a more thorough pre-production review and should prioritise the items by severity and by the cost of the resolution. Most of the resolutions proposed above are incremental (a rule clarification, a UI badge, a configuration flag) and can be implemented without architectural changes, which means they can be folded into the v4.1 release alongside the production deployment.
22. Conclusion
Version 4.0 of the EquiX Definitive Business Logic Document consolidates three prior drafts into a single authoritative specification. It inherits the 25 hard rules and the five-role architecture from v3, resolves all 25 gaps identified during the v3 review, and adds nine new systems: race cancellation, password reset, notifications, audit logging, tie handling, DNF status, Demo Mode, tournament simplification, and verified-account badges. The document also includes the author's original critical analysis in Section 21, which surfaces 13 additional gaps that the team should address before production deployment.
Several scope decisions remain open and are acknowledged rather than concealed. The tournament structure is simplified to a single "final race" model, which preserves the narrative stakes without the complexity of aggregate scoring; a full tournament system is documented as future work. The guess system is intentionally non-financial, with rewards limited to in-kind goods, which keeps the platform clear of gambling regulation. The race simulation is intentionally non-physical, with positions randomised rather than modelled, which keeps the demo deterministic enough to be reproduced reliably. Each of these decisions is a deliberate trade-off that the team made consciously, and each is documented in the relevant section so that future maintainers understand the rationale.
The next steps for the development team are: (1) implement the 34 hard rules as enforced invariants in the Spring Boot backend, with database constraints as the ultimate guarantee and service-layer validation for friendly error messages; (2) implement the 22 sections of this document as ReactJS pages with role-based routing and the visual identity described in Section 1.3; (3) address the 13 items in Section 21 as a v4.1 release before production deployment; (4) conduct a mentor demo using Demo Mode to validate the end-to-end lifecycle; and (5) plan a v5.0 release that addresses the longer-term scope decisions (full tournament system, internationalisation, mobile optimisation) identified as future work throughout this document. With these steps completed, EquiX will be ready for evaluation as the SWP391 capstone deliverable.
